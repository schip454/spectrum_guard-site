# Handoff — перенос на новый VPS и дальнейшие шаги

Дата фиксации: 2026-07-04. Ветка `main`, всё запушено на GitHub
(`schip454/spectrum_guard-site`). Рабочее дерево чистое, локальный HEAD = `origin/main`.

## 1. Что это и где живёт

- Статический сайт на **Astro 5** (SSG). Исходники в репозитории, боевой артефакт — `dist/`.
- **Превью** автоматически публикуется на **GitHub Pages**:
  `https://schip454.github.io/spectrum_guard-site/` (сабпуть — база `/spectrum_guard-site`).
- **Прод** планируется на **reg.ru (host-0)** — статика + PHP-обработчик заявок (форма работает
  только на PHP-хостинге, на Pages не отправляет).

## 2. Перенос на новый сервер — чек-лист

1. `git clone https://github.com/schip454/spectrum_guard-site.git`
2. **Скопировать вручную** секретный файл `server/lead.config.php` со старого сервера
   (он в `.gitignore`, через git НЕ переносится). Если файла нет — создать из
   `server/lead.config.sample.php` и вписать значения (см. раздел 4).
3. Поставить Node и зависимости (раздел 3), собрать: `npm ci && npm run build`.
4. Проверка: `npm run check` (0 ошибок), `npm run build` (31 страница).

Всё остальное (`node_modules/`, `dist/`, `.astro/`, отчёты тестов) регенерируется — не переносим.

## 3. Локальный тулчейн (важно)

На старом сервере **не было системного Node** — он стоял локально в `~/.local`. На новом VPS:

- Нужен **Node 22** (или 18.17+/20.3+). Проще поставить через пакетный менеджер или nvm.
- Команды:
  - `npm ci` — установка зависимостей (нужен `package-lock.json`, он в репо).
  - `npm run dev` — локальный сервер (localhost:4321).
  - `npm run build` — сборка в `dist/` (prebuild сам генерит OG/иконки через sharp).
  - `npm run check` — проверка типов Astro/TS.
  - `npm run test:e2e` — Playwright (нужны системные библиотеки браузера; проще гонять в CI).
- Для превью на Pages сборка идёт с `SITE_BASE=/spectrum_guard-site` + `scripts/rebase-html.mjs`
  (это делает workflow, вручную не нужно).

## 4. Секреты и приём заявок

- Форма → `public/api/lead.php` (PHP) → e-mail + Telegram-бот. Токен НЕ в клиентском JS.
- Секреты — только в `server/lead.config.php` (gitignored):
  - Telegram-бот `@SpectrumGuardBot`, чат — группа «SpectrumGuard | Заявки»,
    `chat_id = -5306758253`. **Доставка проверена** — тестовое сообщение дошло.
  - E-mail для дубля заявок.
- ⚠️ **Токен бота засветился в переписке — сменить через @BotFather**, затем обновить в
  `lead.config.php`. Скрипт для определения chat_id: `TG_TOKEN=… node scripts/tg-chat-id.mjs`.

## 5. Деплой

- **Pages (превью):** `.github/workflows/pages.yml` — авто на каждый push в `main`.
- **reg.ru (прод):** `.github/workflows/deploy.yml` — собирает и льёт `dist/` по FTP.
  Шаг заливки **пропускается, пока не заданы секреты** репозитория:
  `FTP_SERVER`, `FTP_USERNAME`, `FTP_PASSWORD` (+ опц. `FTP_SERVER_DIR`, по умолчанию
  `/public_html/`). Как зададите — деплой включится сам.
  `server/lead.config.php` исключён из FTP-синхронизации (кладётся на хостинг один раз вручную).
- **E2E:** `.github/workflows/e2e.yml` — Playwright/Chromium на 3 разрешениях, авто на push.

## 7. Дальнейшие шаги (roadmap)

1. **Хостинг reg.ru host-0:** задать FTP-секреты в GitHub → автодеплой; один раз залить
   `server/lead.config.php` рядом с `api/`. Проверить сквозную отправку формы → сообщение в
   группу «Заявки».
2. **Сменить токен бота** (@BotFather) и обновить конфиг.
3. **Реальные данные от заказчика:**
   - Реквизиты/лицензия → включить `TRUST_BLOCK_ENABLED` (`src/data/contacts.ts` → `legal`).
   - Цифры-достижения (лет на рынке, выполнено заказов, городов) → добавить блок как у M-Security.
   - Реальный e-mail для заявок, финальные контакты.
4. **Аналитика:** прописать ID Яндекс.Метрики/GA в `src/config.ts` (грузятся только после
   согласия в cookie-баннере).
5. **Яндекс.Бизнес по городам** (офф-сайт) — база для локальной выдачи; после этого можно
   вернуть таргетные гео-страницы с уникальным контентом.
6. **Финальный прогон тестов перед запуском** (UI/UX, smoke, e2e, безопасность) на реальном
   домене.
