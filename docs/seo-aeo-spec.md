# SEO + AEO/GEO спецификация

## Стратегическая рамка (читать до кода)

Сайт работает на два контура одновременно:
- **Классический поиск** (Яндекс, Google) — органические позиции по коммерческим и
  проблемным запросам.
- **Нейровыдача / AEO** (Нейро Яндекса, Алиса, AI Overviews Google, ChatGPT/Perplexity) —
  попадание в ИИ-ответы как цитируемый источник. ИИ строит ответы поверх классического
  топа, поэтому SEO и AEO — одна связка, а не два проекта.

Распределение труда по каналам:
- **Директ** — головные гео-запросы («охрана {город}») здесь и сейчас. Гейт: модерация
  Директа на охранные услуги требует лицензию → платный трафик включается, когда её можно
  показать модератору. До этого момента — только органика и прямые заходы.
- **Органика** — проблемные/длинные запросы (слабо гео-зависимы, низкая конкуренция) +
  города с локальной фактурой (Екб, Москва, СПб).
- **AEO** — асимметричная ставка: ниша слабо оптимизирована под ИИ, есть окно.

## Честные ограничения (не обещать обратное в контенте)

- Без офисов и городских номеров стабильный органический топ-1 по «охрана {город}» в каждом
  городе **не гарантируется** — локальную выдачу Яндекс ранжирует по адресу/телефону/карточке
  Бизнеса. Там, где фактуры нет, голову берёт Директ, органика добирает длинный хвост.
- Анонимный сайт (без лицензии/реквизитов на старте) слабее по E-E-A-T и хуже цитируется ИИ.
  Трастовый блок проектируется заранее и включается флагом, когда данные появятся.

## Гео-стратегия

- **Подпапки**, город в пути: `/{city}/...`. Один домен — весь ссылочный вес в одном месте,
  работает и под Яндекс, и под Google. Поддомены не используем.
- В **Яндекс Вебмастере** задать главный регион (Екатеринбург — есть фактура и приоритет).
  Остальные города — через **Яндекс Бизнес** как «организация с зоной обслуживания»
  (выездное обслуживание, без публичного адреса) по каждому городу, где это проходит модерацию.
- Региональные страницы наполняем **реально локально**, а не подстановкой топонима:
  - Города с `hasLocalProof=true` — местные фото, ориентиры, обезличенные кейсы.
  - Города без фактуры — уникальный текст (не клон), локальный контекст; не плодить тонкие
    дубли (за это Яндекс фильтрует и роняет весь домен).
- Корпоративный мобильный — единый, показываем везде. Городских номеров нет (решение
  заказчика) — компенсируем контентом и карточками Бизнеса.

## Разметка Schema.org (JSON-LD)

| Где | Тип | Ключевые поля |
|-----|-----|---------------|
| Все страницы | `Organization` | name, url, logo, sameAs, contactPoint, areaServed (8 городов) |
| Страницы услуг и гео | `Service` | name, serviceType, provider→Organization, areaServed (город), description |
| Гео-хаб города | `Service`/`LocalBusiness` | areaServed = город; адрес НЕ указываем (его нет) |
| FAQ-блоки | `FAQPage` | вопрос/ответ парами, текст совпадает с видимым на странице |
| Блог | `Article`/`BlogPosting` | headline, author (E-E-A-T!), datePublished, image |
| Везде кроме главной | `BreadcrumbList` | путь навигации |
| Отзывы | `Review`/`AggregateRating` | только реальные, без накруток |

Правила: разметка только для **видимого** контента; валидировать в валидаторе Яндекса и в
Google Rich Results Test; не дублировать сущности.

## AEO / контент под нейровыдачу

- **Answer-First.** Каждая услуговая, гео- и блоговая страница начинается с прямого ответа
  на главный вопрос в 40–60 слов — ИИ «нарезает» и цитирует именно такие блоки.
- **Структура для извлечения.** Чёткие H2/H3, короткие самодостаточные абзацы, списки и
  таблицы сравнения, FAQ-блоки. Каждый абзац — отдельный извлекаемый фрагмент.
- **E-E-A-T.** У статей — указанный автор/эксперт; на сайте — годы работы, обезличенные
  кейсы с цифрами, отзывы. (Реквизиты/лицензия — через флаг, когда появятся.)
- **`llms.txt`** в корне: краткая карта сайта и сути услуг для ИИ-краулеров.
- **Внешний контур (off-site, задача заказчика, не код):** упоминания и материалы на
  авторитетных площадках (VC, Дзен), отзывы в Яндекс Бизнес/2ГИС/картах. ИИ доверяет, когда
  данные на сайте совпадают с внешними источниками (кросс-верификация).

## Роль блога

Блог — опорный слой, не основной канал заявок (эффект копится 3–6 мес.):
1. Ловит проблемные/информационные запросы («что делать, если угрожает бывший», «как
   обеспечить охрану на мероприятии», «законно ли нанять охрану от соседей»).
2. Строит E-E-A-T, который и поисковики, и ИИ используют как фильтр доверия.
3. Кормит нейровыдачу (Answer-First статьи цитируются чаще).
Каждая статья → ведёт на профильную услугу/гео-страницу внутренней ссылкой.

## Перелинковка

- Главная → 5 услуг (через интент-карточки) и → города.
- Хаб услуги → все 8 гео-страниц этой услуги; хаб города → все 5 услуг в городе.
- Гео-страница → соседние услуги в том же городе + та же услуга в соседних городах
  (переключатель города, ссылки индексируемые).
- Блог → профильные коммерческие страницы.

## Технические факторы

- Core Web Vitals в зелёной зоне (LCP < 2.5s, INP < 200ms, CLS < 0.1) — прямой фактор и
  для поиска, и для попадания в ИИ-ответы.
- Чистые ЧПУ, единый trailing slash, 301 на каноникал, без дублей.
- Корректные `title`/`description`/`h1` с топонимом в нужном падеже (из `cities.ts`).
- Sitemap со всеми гео-страницами; robots без случайных запретов индексации.
- Мобайл-фёрст, валидный семантический HTML.

## KPI (раздельно по контурам)

- Классика: позиции по проблемным кластерам и по «охрана {город}» в городах с фактурой;
  органический трафик; число заявок с органики.
- Локалка: видимость карточек Яндекс Бизнес по городам.
- AEO: доля запросов, по которым сайт попадает в ИИ-ответы (фразы из Яндекс Вебмастера, где
  ресурс взят источником) + рост брендовых запросов «Spectrum Guard» в Wordstat. Метрика —
  не позиция, а частота цитирования и брендовый спрос.
- Конверсия: отправки формы, клики в Telegram, заявки на обратный звонок.
